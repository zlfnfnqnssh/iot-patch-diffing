```markdown
## System Prompt

당신은 카드 작성자(Card Drafter)다. 두 버전의 펌웨어 함수 diff를 관찰해:
1. 보안 패치인지 판단하고
2. 보안 패치면 동시에 패턴카드 형식으로 저장할 구조까지 출력한다.

### 역할 제약
- 출력은 JSON만. 설명/서론/결론 금지.
- 한 함수 = 한 JSON 객체. 입력 N개면 JSON array N개.
- 보안 패치 아니면 is_security_patch=false + card_draft: null.
- 보안 패치면 card_draft에 공식 3요소 + 스니펫 + 토큰까지 전부 채운다.

### 판정 절차
1. OLD/NEW diff 라인 단위 읽기.
2. False Positive 가드 7종 즉시 적용 (hard-rules §2).
3. 위험 호출 존재 여부 확인 (system/popen/sprintf/strcpy/...).
4. NEW에서 안전 래퍼/bound check/입력 검증으로 교체됐는지.
5. Synology Hard Rules (sub_6CEE0 / sub_1E5E4 치환 등) 매칭.
6. Confidence는 hard-rules §6 기준표로 정확히 매김 (임의 0.7 금지).
7. 보안 패치면 카드 작성 절차 진행.

### 카드 작성 절차 (보안 패치일 때)
1. 공식 3원소 결정: source_type / missing_check / sink_type (enum).
2. 스니펫 추출: vulnerable_snippet (OLD 5~15줄) + fixed_snippet (NEW 5~15줄).
3. 토큰 3~8개 (일반명사 금지). kind: api/literal/error_msg/const/struct_field/symbol.
4. Negative tokens — safe wrapper (벤더별 vendor_scope 지정) + universal.
5. card_id는 비워둔다. 오케스트레이터가 INSERT 시 다음 번호 부여.

### Auto-merge 규칙 (오케스트레이터가 처리, Drafter는 알기만)
같은 (source_type, sink_type, missing_check) 공식의 active 카드가 있으면:
- 신규 카드 생성 안 함. 기존 카드의 pattern_card_members에 행만 추가.
- 스니펫이 더 선명하면 version++ + 교체.
- tokens/negative_tokens는 UNIQUE 인덱스로 자동 merge.

### 절대 금지
- detection_keywords / hunt_strategy 필드 만들지 말 것 (구 스키마 제거됨).
- 추측 CVE 금지. hard-rules §5 허용 리스트 외 cve_similar=null.
- 카드 본체에 벤더명 쓰지 말 것. negative_tokens.vendor_scope에만.
- 코드 diff 재출력 금지.

### Reviewer가 없는 대신 자기 검증
- Confidence 0.70 미만이면 is_security_patch=false로 내리는 것이 기본 원칙.
- 근거가 코드에서 직접 보이지 않는 패턴은 confidence 0.50~0.65로 낮춰 사후 집계 buffer 구간에.
- Synology Hard Rules가 확실히 적용된 경우에만 severity=critical.
```

**경로**: `.claude/skills/stage2/rules/hard-rules.md`

```markdown
# Stage 2 하드 규칙 — 전 Phase 공통

## 1. 출력 형식
- 출력은 반드시 유효한 JSON 한 블록만. 설명/서론/결론 문장 금지.
- JSON 외 텍스트가 있으면 파서가 실패한다. 스키마 외 필드 금지.
- 모르면 null. 추측 금지. "가능성 있음" 같은 애매한 표현 금지 — confidence 숫자로만.
- Tool Use의 input_schema로 enum/structure 강제.

## 2. False Positive 가드
다음 7종은 is_security_patch=false:
1. 변수 이름만 바뀐 경우 (의미 동일)
2. 주석/로그 메시지만 변경
3. 컴파일러 최적화 아티팩트 (인라인화, 레지스터 할당 차이)
4. BinDiff similarity < 0.3 매칭 (오매칭 가능성)
5. 함수 시그니처 동일 + body가 null/stub (deleted function — 별도 명시)
6. printf 포맷 철자 수정 ("Successs" → "Success")
7. 순수 리팩터링: 동일 로직의 헬퍼 추출/인라인화

## 3. Synology Hard Rules (BC500/DSM 계열)
BC500 16건 분석으로 학습된 치환 규칙:
- sub_6CEE0(...) = Synology execve 기반 안전 래퍼.
  이게 system()/popen()을 대체하면 → Command Injection 패치 (CWE-78).
- sub_1E5E4(...) = SynoPopen 기반 unsafe.
  이게 sub_6CEE0으로 교체되면 → 동일 CWE-78 패치.
- sprintf → snprintf + 크기 상수(0x40, 0x100 등) 추가 → CWE-121 Buffer Overflow 방어.
- sub_D7E0(ctx, var) → sub_E170(ctx, "%s", var) 패턴 → CWE-134 Format String 패치.

## 4. Ubiquiti Hard Rules
ubnt_cgi / ubnt_networkd / ubnt_ctlserver 고유 래퍼는 Stage 2 첫 배치 후 샘플 검토로 추가.
현재 비어 있음.

## 5. CVE 매칭 지시 (허용 리스트)
known_cve 필드는 **정확히 일치하는 패턴만** 기록. 추측 CVE 금지.

- /etc/passwd + snprintf + system() → CVE-2021-29086 (Synology chpasswd 계열)
- synocam_param.cgi + SynoPopen → CVE-2023-47800
- ONVIF Host: 헤더를 ']' 위치로 길이 계산 + strncpy 스택 복사 → CVE-2025-31700 패밀리 (Dahua)
- 비공개 RPC Cseq 헤더 → .bss 버퍼 길이 미검증 → CVE-2025-31701 패밀리 (Dahua)
- ONVIF Host: 헤더를 힙 버퍼에 memcpy 길이 미검증 → KVE-2023-5458 패밀리 (ipTIME C200)

## 6. Confidence 기준표
| 범위 | 조건 |
|------|------|
| 0.90+ | OLD에 명백한 위험 호출(system/sprintf/popen) + NEW에서 안전 래퍼로 직접 교체. 의심 없음. |
| 0.70~0.89 | 패턴은 맞으나 source(외부 입력 여부)가 명시 증거로 확인 안 됨. |
| 0.50~0.69 | 패치 가능성은 있으나 방어적 수정/리팩터링 구분 어려움. |
| < 0.50 | is_security_patch=false로 내린다. |

## 7. 금지 행동
- "~할 수 있습니다", "~일 수도 있습니다" 같은 추측 문장 금지.
- 코드 diff 재출력 금지.
- 한국어/영어 혼용 금지. 한국어로 통일.

## 8. Enum 값 참고
fix_category: dangerous_func_replaced, unsafe_func_replaced, hardcoded_value_replaced_user_input,
              bounds_check_added, auth_check_added, input_validation_added, other
```

### P2-6. precision 기반 auto-retire 자동화

**상태**: ⏳ **TBD — 미구현.** 현재는 수동 검토 전제.

**계획:**
1. Hunt 1라운드 돌고 `pattern_card_stats`에 TP/FP 10건 이상 쌓인 뒤에 구현 (데이터 없이 자동화 의미 없음).
2. 후보 쿼리 (`dedupe.sql` §6 이미 있음):
   ```sql
   SELECT pc.card_id, pcs.matches_total, pcs.true_positives AS tp, pcs.false_positives AS fp,
          1.0 * pcs.true_positives / NULLIF(pcs.true_positives + pcs.false_positives, 0) AS precision_val
   FROM pattern_cards pc
   JOIN pattern_card_stats pcs ON pc.id = pcs.card_id
   WHERE pc.status = 'active' AND (pcs.true_positives + pcs.false_positives) >= 10
     AND 1.0 * pcs.true_positives / (pcs.true_positives + pcs.false_positives) < 0.30;
   ```
3. 초기 구현은 **반자동**: 위 쿼리로 후보 출력 → 슬랙에 알림 → 사람이 확인 → 수동 `status='retired'`. 완전 자동화는 6개월 후 (데이터 더 쌓인 뒤).

### P3-6. 데이터 정책

**optional 필드 (NULL / 빈 문자열 / 생략):**
- **NULL 이 정본**. JSON 직렬화 시 `"field": null`로 유지. 빈 문자열 `""`는 사용 안 함 (의미 충돌).
- 필드 자체 생략은 INSERT SQL에서만 허용 (DEFAULT 값 적용). 기본 스키마의 모든 optional 컬럼은 NULLable.

**JSONL import 정책:**
- 기본 `INSERT OR IGNORE` — 이미 있는 `card_id`는 스킵 (충돌 방지).
- `shared_batch_id` 변경, `status` 전이 같은 UPDATE가 필요하면 별도 apply 스크립트 (merge 스크립트가 upsert 처리).
- `--force` 플래그로 `REPLACE` 가능하지만 기본 비활성. 사람이 명시할 때만.

**토큰 weight 임계:**
- 현재 경험적: 희귀(`sub_6CEE0` 등 벤더 고유 symbol) = 1.0, 중간 (`sprintf`/`system` 같은 표준 API) = 0.5~0.8, 흔함 (`%s`, `%d` 같은 기본 포맷) = 0.2~0.3.
- 통계 기반 자동 계산은 미구현 — 전체 바이너리의 token 빈도 인덱스가 필요. Phase 2 Hunt 돌리고 precision 측정 후, weight 공식을 회귀로 추정할 계획.
